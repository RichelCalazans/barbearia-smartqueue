# Plano: Vitest e Estrategia de Testes

## Objetivo

Criar uma base rapida de testes unitarios para proteger mudancas em regras de negocio antes das Cloud Functions e da refatoracao do dashboard.

## Estado Atual

- `package.json` nao possui `vitest`.
- O script `lint` roda `tsc --noEmit`.
- A aplicacao usa React 19 + Vite 6.
- Ha logica testavel em services e utils, especialmente `TimePredictorService`, `ConfigService`, `AnalyticsService` e `src/utils.ts`.
- Component tests do `BarberDashboard.tsx` devem esperar a refatoracao, porque o componente tem 1883 linhas e muito estado acoplado.

## Decisao

Usar Vitest com mocks via `vi.mock`, sem Firebase Emulator neste primeiro ciclo. O objetivo e feedback rapido e baixo custo de setup.

## Dependencias

Instalar:

```bash
npm install -D vitest happy-dom @testing-library/react @testing-library/user-event @testing-library/jest-dom
```

`happy-dom` entra desde o inicio para permitir testes leves de hooks/componentes depois, mesmo que a primeira leva foque funcoes puras.

## Setup

- Adicionar scripts:
  - `test`: `vitest run`
  - `test:watch`: `vitest`
  - `test:coverage`: `vitest run --coverage`
- Configurar `test` em `vite.config.ts`:
  - `environment: 'happy-dom'`;
  - `setupFiles: './src/test/setup.ts'`;
  - aliases iguais aos do Vite, se houver.
- Criar `src/test/setup.ts` com matchers e mocks globais minimos.
- Criar helpers em `src/test/factories.ts` para `AppConfig`, `AppState`, `Client`, `QueueItem` e `Attendance`.

## Top 5 Alvos

1. `TimePredictorService`
   - cliente novo usa multiplicador;
   - cliente recorrente usa media historica;
   - soma de servicos respeita `tempoBase`;
   - buffer e arredondamento nao geram tempo negativo.

2. `src/utils.ts`
   - `isDateEnabled`;
   - `getAvailableDates`;
   - `getScheduleForDate`;
   - dias desabilitados e sabado com horario diferente.

3. `ConfigService`
   - `deriveBarberStatus`;
   - `isOutsideScheduledWindow`;
   - nova funcao pura `decideAgendaAction(config, state, now)` quando o plano de Cloud Functions comecar.

4. `AnalyticsService`
   - extrair ou expor funcoes puras para calculo de utilization, aderencia e agregacoes por periodo;
   - testar primeiro a logica pura, nao queries Firestore.

5. Validacao
   - criar helpers para telefone, email, horario `HH:MM` e datas;
   - usar esses helpers depois em forms e services.

## Mocks

- Mockar `firebase/firestore` por modulo quando testar services que importam Firestore.
- Preferir testar funcoes puras extraidas antes de testar metodos que fazem I/O.
- Evitar Emulator neste ciclo para manter setup curto.

## Criterios de Aceite

- `npm run lint` passa.
- `npm run build` passa.
- `npm run test` passa em ambiente local.
- Pelo menos 20 testes unitarios cobrindo os cinco alvos acima.
- Documentar em `docs/PADROES.md` como criar novos testes.

## Sequencia Recomendada

1. Instalar dependencias e scripts.
2. Configurar Vitest.
3. Adicionar factories.
4. Testar `src/utils.ts` e `ConfigService.deriveBarberStatus`.
5. Testar `TimePredictorService`.
6. Extrair funcoes puras de analytics/agenda conforme necessidade.
7. So depois iniciar testes de componentes pequenos.
